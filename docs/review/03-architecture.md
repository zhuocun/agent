## 1. New ideas & opportunities / better options (not in existing docs)

### 1.1 AI SDK v6 is GA — the "v5 vs v6" open question is effectively resolved
- **What's new:** AI SDK **v6 went GA on 2025-12-22**; current npm is **`ai@6.0.x`** (≈6.0.190 as of late May 2026). v5 shipped July 2025; v6 is now the default major. **[Verified]** ([vercel.com/blog/ai-sdk-6](https://vercel.com/blog/ai-sdk-6), [github.com/vercel/ai/releases](https://github.com/vercel/ai/releases), [npm @ai-sdk/vercel](https://www.npmjs.com/package/@ai-sdk/vercel))
- **Why it matters:** The research doc and PRD repeatedly hedge "v5/v6 era — [confirm at build: v5 vs v6]." That open question is now answerable: **target v6.** v6 brings a first-class **`ToolLoopAgent`/Agent** abstraction (typed stop conditions `stepCountIs()`/`hasToolCall()`), **tool-execution approval (`needsApproval`)** for human-in-the-loop, stable **MCP** (OAuth, resources, prompts, elicitation), native **`rerank()`**, **DevTools**, and unified tool-calling + structured output (standalone `generateObject`/`streamObject` are **deprecated** — confirmed in the official v6 migration guide — in favor of `streamText`/`generateText` with `Output.object()`). **[Verified]** ([official v6 migration guide](https://ai-sdk.dev/docs/migration-guides/migration-guide-6-0); [digitalapplied AI SDK 6 deep dive](https://www.digitalapplied.com/blog/vercel-ai-sdk-6-deep-dive-features-tool-calls-2026))
- **Recommendation:** Lock the PRD to **AI SDK v6** as the build target (keep the "isolate behind adapter / read migration notes" discipline). Use the v6 `Agent` primitive for the P1 tool/agent loop rather than hand-rolling multi-step. Note `generateObject`/`streamObject` deprecation now, since the MVP ships "structured outputs + schema validation" (PRD 00 §5) — write that against the v6 `Output.object()` API, not the deprecated helpers. There is also early signal of **AI SDK v7 / `WorkflowAgent`** tied to Vercel Workflows (see 1.3) — treat v7 as horizon, not build target. **[Verified for v6; v7 Uncertain]**

### 1.2 Resumable streams: the stop/abort model is more subtle than the docs assume — needs an explicit out-of-band cancel
- **What's new / corrected:** In v5/v6 resumable-stream mode, **client-side aborts (calling `stop()`, refreshing, closing the tab, losing network) are all treated as disconnects and DO NOT cancel the underlying generation by design** — the server keeps generating so the client can reconnect. To actually stop generation you must build a **dedicated stop endpoint** that persists the partial message, cancels the producing work, and clears `activeStreamId`. **[Verified]** ([ai-sdk.dev resume-streams docs](https://ai-sdk.dev/docs/ai-sdk-ui/chatbot-resume-streams), [vercel/ai #8390](https://github.com/vercel/ai/issues/8390), [#6502](https://github.com/vercel/ai/issues/6502))
- **Why it matters:** This is **bigger than the PRD's framing.** PRD §5.1 frames the orphaned-run risk as an edge case ("a stop issued from a *resumed* connection may not reach the original process"). In reality, **once resumable streams are on, the *normal* stop path also won't cancel generation** — every stop is a disconnect. So the orphaned-run mitigation (Stream-row source-of-truth + Redis abort channel + reaper) is not an edge-case nicety; **it is the primary stop mechanism** and must exist the moment resumable streams ship. There's also a known open bug where resume fails if the resumed stream starts with a `text-delta` for an existing part ([vercel/ai #13160](https://github.com/vercel/ai/issues/13160)) and tab-backgrounding resume issues ([#11865](https://github.com/vercel/ai/issues/11865)). **[Verified]**
- **Recommendation:** Promote the stop endpoint to a **P0-with-resumable-streams** design item. Since the MVP's headline UX is "**stop-that-preserves-partial-output**" (PRD 00 §5) but resumable *replay* is deferred to P1, decide explicitly: **MVP (no resumable replay) → `AbortSignal` cancels generation normally** (simple, correct); **the moment resumable replay lands (P1) → stop must switch to the out-of-band cancel endpoint.** Document this as a behavioral switch, because the abort semantics literally invert when resumability turns on. The `resumable-stream` package is Redis-pub/sub and works with **Upstash Redis** via its generic client interface (one `INCR` + `SUBSCRIBE` per stream; minimal Redis cost unless a resume actually happens), so the Upstash choice is validated. **[Verified]** ([github.com/vercel/resumable-stream](https://github.com/vercel/resumable-stream), [Upstash resumable LLM streams](https://upstash.com/blog/resumable-llm-streams))

### 1.3 Vercel Workflows / Durable Execution — a new answer to the central "long stream vs timeout" risk
- **What's new:** Vercel shipped **Workflows** — managed durable execution for TS/JS/Python that **checkpoints state at every `await` boundary**, survives timeouts/deploys/crashes, can pause for "minutes to months," and provides **durable streams** (`getWritable()` gives a persistent stream multiple clients can connect/disconnect/reconnect/resume). It integrates with the AI SDK; `DurableAgent`/`@workflow/ai` turns each LLM call + tool invocation into its own durable step, each getting the full function timeout with automatic retries. **[Verified]** ([vercel.com/blog/a-new-programming-model-for-durable-execution](https://vercel.com/blog/a-new-programming-model-for-durable-execution), [vercel.com/docs/workflows](https://vercel.com/docs/workflows), [github.com/vercel/workflow](https://github.com/vercel/workflow))
- **Why it matters:** This is a materially better primitive than "Fluid Compute keeps the function alive longer + resumable-stream Redis layer" for **long agentic/multi-step runs** (the P1 tool/agent loop). It collapses "resumable streams" + "background job" + "reaper" into one durable-step model, and it solves the orphaned-run problem structurally (steps are checkpointed and individually retried/cancelled). It is not in the research doc or PRD at all.
- **Recommendation:** Add Vercel Workflows to the **architecture decision radar as the P1 path for the agentic/tool loop and any multi-minute generation**, replacing the bespoke "QStash reaper + resumable-stream + Stream table" reconciliation for those flows. For the **text-core MVP**, the simpler streamText + (deferred) resumable-stream path is still fine — don't pull Workflows forward unnecessarily. But the PRD's "Risk #1: long streams vs timeouts" mitigation table should list Workflows as the strategic answer for P1 agent loops. **Caveat:** Workflows is Vercel-proprietary; it cuts against the PRD's "no one-way doors" principle, so gate it behind the same provider/runtime adapter discipline (Temporal/Restate + AI SDK are the portable equivalents — both have shipped AI SDK integrations). **[Verified]** ([temporal.io durable agents](https://temporal.io/blog/building-durable-agents-with-temporal-and-ai-sdk-by-vercel), [restate.dev](https://www.restate.dev/blog/building-durable-agents-with-vercel-and-restate))

### 1.4 Next.js 16 is GA — Cache Components/PPR shipped, and `middleware.ts` → `proxy.ts`
- **What's new:** **Next.js 16 shipped 2025-10-21** (GA). PPR is no longer "graduating in 16" (research §2 future tense) — it's **stable, reframed as "Cache Components"** via `cacheComponents: true` + the `"use cache"` directive (opt-in caching; everything dynamic by default). Turbopack is the default bundler (stable). React Compiler support is stable (off by default). **`middleware.ts` is deprecated and renamed to `proxy.ts`, which runs on the Node.js runtime** (the old Edge `middleware.ts` is deprecated, removal later). React 19.2. Min Node 20.9; `params`/`searchParams`/`cookies()`/`headers()` are now **async** (await). **[Verified]** ([nextjs.org/blog/next-16](https://nextjs.org/blog/next-16))
- **Why it matters:** Two concrete impacts. (a) The PRD/research treat PPR as a future "graduating" feature — it's here now and is the **right default rendering model** for the static-shell-instant + streamed-chat-pane app. (b) **The auth/rate-limit-in-middleware pattern moves to `proxy.ts` on Node runtime** — this affects where the PRD's "Upstash rate limit (user + IP)" interceptor lives and quietly removes a reason to reach for the Edge runtime. The async `cookies()/headers()` change touches every Better Auth session read in route handlers.
- **Recommendation:** Update the PRD to (a) name **Next.js 16 + Cache Components** as the rendering baseline and adopt `"use cache"` for the conversation-list/model-registry hot reads (pairs with the Upstash cache layer); (b) put edge-gating/rate-limit/auth-guard logic in **`proxy.ts` (Node runtime)**, and drop any lingering "edge middleware" assumption; (c) note the async-dynamic-API migration as a build gotcha. **[Verified]**

### 1.7 Langfuse SDK v4 is now OTel-native — collapses the "two systems to wire" tradeoff
- **What's new:** **Langfuse SDK v4 is a thin layer on top of the official OpenTelemetry client** — it auto-converts emitted OTel spans into Langfuse observations and adds LLM helpers (token usage, cost, prompt linking, scoring). OTel **GenAI semantic conventions** exited experimental for **client spans in early 2026** (metrics/full spec still "Development"). Datadog/Grafana now ingest these natively. **[Verified]** ([langfuse.com/integrations/native/opentelemetry](https://langfuse.com/integrations/native/opentelemetry), [opentelemetry.io GenAI semconv](https://opentelemetry.io/docs/specs/semconv/gen-ai/), [opentelemetry.io/blog/2026/genai-observability](https://opentelemetry.io/blog/2026/genai-observability/))
- **Why it matters:** The PRD lists "two systems to wire" as the Langfuse+OTel tradeoff. With SDK v4 it's effectively **one instrumentation surface (OTel) that Langfuse consumes** — lower wiring cost and stronger vendor-portability than the doc implies. **But** there's a hidden cost: **self-hosted Langfuse now requires ClickHouse + Postgres + Redis/S3** (not a single container) — so "self-hostable, free" is heavier than the research's "free Hobby tier ~50k events/mo" framing suggests. **[Verified]** ([langfuse.com/self-hosting](https://langfuse.com/self-hosting))
- **Recommendation:** Standardize instrumentation on **OTel GenAI semantic conventions** and treat Langfuse as the OTel-consuming UI (cloud tier for MVP; self-host only if data-residency demands it, budgeting for the ClickHouse stack). Pin to GenAI-semconv **client-span** attributes (stable-ish) and avoid depending on still-experimental metric conventions. Update PRD §5.6 tradeoff line.

### 1.8 Token/cost-budget rate limiting + circuit breakers is the 2026 standard (request-count sliding window alone is insufficient)
- **What's new:** 2026 best practice for LLM apps is **multi-layered, token-aware rate limiting**: request-count limits *plus* **token-based limits** *plus* **per-user cost ceilings/budgets** *plus* **circuit breakers** that trip when an agent loops on an expensive tool ("financial DoS"). Maps to **OWASP LLM10:2025 — Unbounded Consumption.** **[Verified]** ([zuplo token-based rate limiting](https://zuplo.com/learning-center/token-based-rate-limiting-ai-agents), [oneuptime LLM rate limiting](https://oneuptime.com/blog/post/2026-01-30-llm-rate-limiting/view), [genai.owasp.org LLM Top 10 2025](https://genai.owasp.org/llmrisk/llm01-prompt-injection/))
- **Why it matters:** PRD §5.6 specifies sliding-window request limits "weighted by model cost class" — a good instinct but still **request-count-centric**. The real exposure (the "inference whale" risk in PRD 05 §3 and PRD §8 Risk #5) is **token/$ spend**, which a request-count window doesn't bound (one request can be 200k tokens). The Gateway's budget feature helps, but the app needs its own per-user **token + dollar budgets** and a **circuit breaker** for runaway agent loops (becomes critical when the P1 tool loop lands).
- **Recommendation:** Upgrade PRD §5.6 to require **(a) request-rate limits, (b) token-rate limits, (c) per-user rolling cost-budget caps enforced from the `message.cost_usd` ledger, and (d) a circuit breaker on multi-step/tool runs.** Tie these explicitly to OWASP **LLM10 Unbounded Consumption**. This also makes the §6 `message.cost_usd` field do double duty as the live budget-enforcement signal, not just a transparency display.

## 3. PRD 04 review — gaps, errors, outdated items, inconsistencies, risks

Ordered roughly by severity. Section references are to `docs/prd/04-technical-architecture.md`.

### 3.1 [ERROR / compliance — high] EU AI Act dates are likely wrong and the deferral is risky (§5.7, also PRD 00 §10)
- **Issue:** §5.7 splits Article 50 into **AI-interaction disclosure (~Aug 2026, P0)** and **AI-content marking (~Dec 2, 2026, P1)**. Research indicates **both obligations apply from 2 August 2026 under the same Article 50** ([artificialintelligenceact.eu/article/50](https://artificialintelligenceact.eu/article/50/)). The Dec-2-2026 date does not match Article 50's content-marking timing; it may be a confusion with a Code-of-Practice milestone or GPAI obligations.
- **Risk:** Treating machine-readable AI-content marking as P1 ("enforcement P1") could mean **shipping non-compliant in the EU** for products generating synthetic text/image, given the privacy-conscious-EU persona (PRD 00 §4). The product *generates* text (and image at P2), squarely within Article 50(2).
- **Fix:** Re-verify both dates against the first-party Article 50 text and the final Code of Practice (final expected ~June 2026). If both are 2 Aug 2026, **pull content-marking forward to P0-design / P1-enforce-before-EU-launch** and align PRD 00 §10. The schema hooks (`message.ai_generated`, `document.ai_generated`) are good — but note that **Article 50 wants machine-readable marking in the *output artifact*** (e.g., C2PA-style content credentials for images, watermark/metadata for text/exports), not just a DB boolean. Add an explicit "output-embedded marking" requirement, especially for exports and any P2 image generation. **[Verified on Aug-2026; confirm exact content-marking modality against final Code of Practice]**

### 3.2 [ERROR / stale] "[confirm at build: v5 vs v6]" is now resolvable — and structured-output API choice is affected (§4, §5.1, §8 Risk #3, §9 Q6)
- **Issue:** Multiple places hedge v5-vs-v6. v6 is GA. The PRD should commit to **v6**.
- **Knock-on:** PRD 00 §5 puts "**structured outputs + schema validation**" in the P0 MVP. In v6, **`generateObject`/`streamObject` are deprecated** ([official v6 migration guide](https://ai-sdk.dev/docs/migration-guides/migration-guide-6-0)) in favor of `streamText`/`generateText` + `Output.object()`. Building the MVP against the deprecated API is avoidable rework.
- **Fix:** Change the stack table and risks to target **AI SDK v6**; rewrite the structured-output requirement against the v6 `Output` API; use the v6 **`Agent`/`ToolLoopAgent`** primitive for the P1 tool loop instead of hand-rolled multi-step. Keep the "isolate behind adapter, read migration notes" discipline (now pointed at v6→v7). **[Verified]**

### 3.3 [GAP / risk — high] The stop/abort semantics inversion under resumable streams is under-specified (§5.1)
- **Issue:** §5.1 treats the orphaned-run problem as an edge case from a *resumed* connection. Per §1.2 above, **with resumable streams ON, every stop/refresh/close is a disconnect that does NOT cancel generation** — so the out-of-band cancel is the *primary* path, not an edge case. Conversely, the MVP ships **without** resumable replay (P1), where a normal `AbortSignal` *does* cancel — so the abort behavior **inverts** between MVP and P1.
- **Risk:** If built naively, enabling resumable streams in P1 silently breaks the headline "stop-that-preserves-partial-output" UX and re-introduces billing-until-timeout for *every* stop, not just resumed ones.
- **Fix:** Document two explicit modes in §5.1: **(MVP, no replay)** `AbortSignal` cancels generation; persist partial on abort. **(P1, replay on)** stop is a dedicated server endpoint that persists partial + cancels the producing work + clears `activeStreamId` + publishes abort on the Redis channel; the reaper backstops. Call out that abort semantics change when resumability is enabled. Reference [vercel/ai #8390](https://github.com/vercel/ai/issues/8390) and the known resume bugs [#13160](https://github.com/vercel/ai/issues/13160)/[#11865](https://github.com/vercel/ai/issues/11865) as build-time watch items. Consider **Vercel Workflows / DurableAgent** for the P1 agent loop, which handles this structurally (§1.3).

### 3.4 [GAP — high] Rate limiting is request-count-centric; the real exposure is token/$ spend (§5.6, §8 Risk #5)
- **Issue:** §5.6 specifies sliding-window limits "weighted by model cost class." That bounds request *rate*, not token/$ *consumption* — but the documented business risk (PRD 05 §3 inference whales; PRD §8 Risk #5) is dollars. 2026 best practice = **token-rate limits + per-user cost-budget ceilings + circuit breakers** (OWASP **LLM10 Unbounded Consumption**).
- **Fix:** Require, in §5.6: (a) request-rate limit, (b) token-rate limit, (c) per-user rolling **cost-budget cap** enforced off the `message.cost_usd` ledger (especially for guests/free tier), (d) **circuit breaker** for runaway multi-step/tool loops, (e) Gateway budgets as a second layer. Map explicitly to OWASP LLM10. This makes `message.cost_usd` an enforcement signal, not just a display value. **[Verified best practice]**

### 3.5 [GAP — medium] No data-deletion/cascade design for object storage, Redis, traces, or the provider/Gateway side (§5.7)
- **Issue:** §5.7 requires one-click **delete (hard-delete user + cascade)** and **configurable retention**. The §6 schema cascades within Postgres, but the PRD does not address **deleting/expiring**: (a) **attachments in R2/Blob** (orphaned objects after row delete), (b) **buffered stream data + cached reads in Redis**, (c) **traces/PII in Langfuse/OTel** (these can contain full prompt/response content — a real GDPR surface), (d) **logs at the Gateway/provider** (ZDR helps here — tie it in). "Hard-delete user + cascade" in Postgres alone is **not** GDPR-complete.
- **Fix:** Add a cross-store deletion requirement: a delete job (QStash) that fans out to object storage, Redis keys, and the observability backend (Langfuse delete API / OTel retention policy), plus a documented retention TTL per store. Note that **observability traces are a PII store** and must honor no-telemetry mode (PRD §5.7 mentions the mode but not that traces are in scope for export/delete).

### 3.6 [GAP — medium] No migration / data-portability story; "no one-way doors" is asserted but not proven for the riskiest dep (auth) (§5.5, §5.8, §8 Risk #2)
- **Issue:** The "thin adapter" portability story is strong for provider/storage/cache/queue/DB. But **auth is the hardest thing to migrate** (password hashes, sessions, OAuth links, the anonymous→linked graph) and the PRD's own §9 Q1 marks the auth choice as unresolved and gated on a spike. Meanwhile Better Auth is now clearly the right answer (Auth.js is patch-only — §2-F), which actually *de-risks* this if committed.
- **Fix:** (a) Given §2-F, **commit to Better Auth** in the stack table (not "lead recommendation / or Supabase / or Auth.js fast start") and remove Auth.js as a fallback — it's now a deprecated path. Keep Supabase Auth as the *only* alternative, contingent on choosing Supabase as the data platform. (b) Keep the §9 Q1 spike but reframe it as "prove guest→link with no data loss **in Better Auth**," not "which auth." This also resolves the latent inconsistency where PRD 00 §8 / §11-D4 already name Better Auth as *the* choice while PRD 04 still presents it as one of three.

### 3.7 [INCONSISTENCY — medium] BYOK is P0 but `attachment`/storage is P1 — yet BYOK metering needs the cost ledger that depends on usage capture only, OK; but guest+BYOK interaction is unspecified (§5.2, §5.5, §6)
- **Issue 1:** §5.2 says "BYOK usage is still rate-limited for abuse but not metered for billing," and §5.5 says guests are rate-limited by IP. **Can a guest (anonymous, no account) add a BYOK key?** The `api_key` table FKs `user_id` (which exists for guests since they get a real row), so technically yes — but that's a key-security and abuse question (an anonymous user storing a provider key; key tied to an evictable guest identity that may never upgrade). Unspecified.
- **Issue 2:** §6 `api_key` has **no `key_hint` uniqueness / one-key-per-provider-per-user constraint**, no `last_used_at`, no rotation/`updated_at`, and no soft-delete — operationally thin for a P0 security-sensitive table.
- **Fix:** (a) State the policy: BYOK requires a **non-anonymous** account (recommended), or explicitly allow-and-bound it. (b) Add to `api_key`: unique `(user_id, provider)` (or named keys), `last_used_at`, `updated_at`, and a revocation/rotation flow; consider storing only the wrapped DEK + ciphertext per envelope-encryption best practice (a fresh DEK per record, AES-256-GCM — §1 BYOK research). **[Verified envelope-encryption pattern]** ([Google KMS envelope encryption](https://docs.cloud.google.com/kms/docs/envelope-encryption))

### 3.8 [GAP — medium] Postgres is the only system of record, but several stores hold authoritative-ish state without a consistency story (§5.1, §6)
- **Issue:** The `stream` table has a **unique partial index `(chat_id) where status='active'`** (at most one active stream per chat). Good. But the **resumable-stream buffer lives in Redis**, and Redis is explicitly best-effort/evictable (PRD §5.1 acknowledges retention limits). The reconciliation between "Postgres says active" and "Redis buffer evicted" (reaper marks aborted) is described, but **what the user sees** (a half-streamed message whose Redis buffer is gone on reconnect) isn't specified. Also: concurrent sends to the same chat from two devices both trying to create an `active` stream will hit the unique index — what's the UX/error?
- **Fix:** Specify the degraded-path UX (on reconnect with no buffer: show persisted partial + "generation interrupted, retry"). Specify behavior when the unique-active-stream constraint is violated (e.g., 409 + "a response is already generating in this chat"). Minor, but these are the rough edges that bite in production.

### 3.9 [OUTDATED — low/medium] Rendering model and middleware are pre-Next.js-16 (§3, §4, §5.8)
- **Issue:** The PRD doesn't mention **Next.js 16 / Cache Components / PPR-stable**, and any implied **Edge middleware** for rate-limit/auth is now `proxy.ts` on Node (§1.4). The reference diagram and §5.1's "Node runtime" choice are correct, but the rendering/edge story is implicitly pre-16.
- **Fix:** Name Next.js 16 + Cache Components as the baseline; adopt `"use cache"` for hot reads (conversation list, model registry) layered with Upstash; move rate-limit/auth-guard interception to `proxy.ts`; note async `cookies()/headers()`/`params` as a build gotcha for Better Auth session reads. **[Verified]**

### 3.10 [GAP — low] Observability cost/operational footprint understated; OTel GenAI conventions not named (§5.6, §4)
- **Issue:** §5.6 frames Langfuse + OTel as "two systems to wire." Per §1.7, Langfuse v4 *is* OTel-native (one surface), **but self-hosting it now needs ClickHouse+Postgres+Redis** — a non-trivial ops footprint the PRD doesn't flag. Also the PRD doesn't anchor on the **OTel GenAI semantic conventions** as the schema for spans (model id, token counts, cost) — which would make the "export to any APM" claim concrete and align with the §6 per-message usage contract.
- **Fix:** Standardize on **OTel GenAI semantic conventions** for LLM spans; treat Langfuse as the OTel consumer (cloud tier for MVP; self-host only with the ClickHouse-stack budget acknowledged). Pin to stable-ish **client-span** attributes; treat metric conventions as still-experimental. **[Verified]**

### 3.11 [GAP — low] Multi-device / multi-tab SSE connection limit not considered (§5.8)
- **Issue:** §5.8 defers true multi-device sync (fine). But even the *single-device, multi-tab* case can hit the **HTTP/1.1 6-connections-per-domain SSE cap** (§2-D), and CDN/proxy buffering can defeat token streaming. Not mentioned.
- **Fix:** Note HTTP/2 (Vercel serves HTTP/2+) mitigates the connection cap; add a build check that streaming responses set no-buffering headers and the platform doesn't buffer SSE. Low risk on Vercel but a classic gotcha behind corporate proxies / on some mobile carriers. **[Verified]**

### 3.12 [STRENGTH — worth keeping] Things the PRD gets right
- The **per-message model+token+cost contract** in `message` (with wide `numeric(14,8)`) is exactly right and forward-thinking; it's also the budget-enforcement signal (§3.4).
- The **`stream` table + unique-active-partial-index + reaper** design is the correct shape; it just needs the abort-semantics clarification (§3.3).
- The **adapter-everywhere portability** principle is sound; the only weak link is auth (§3.6), now resolvable.
- **`is_anonymous` + Better Auth anonymous→link** is the right call and is validated by current sources.
- The honest **[confirm at build]** tagging is excellent practice — this review mostly just resolves those tags.

## 4. Top 5 recommendations (prioritized, actionable)

1. **Fix the EU AI Act compliance plan (highest risk, cheap to fix now).** Re-verify Article 50 dates against first-party text; both interaction-disclosure and machine-readable AI-content marking most likely apply **2 Aug 2026** (not Dec). Pull content-marking to **P0-design / pre-EU-launch enforce**, and require marking embedded in the *output/exported artifact* (C2PA-style for images, metadata for text), not just the `ai_generated` DB flag. (§3.1)

2. **Commit the now-resolved version decisions and remove dead alternatives.** Target **AI SDK v6** (use `Agent`/`Output.object()`; drop deprecated `generateObject/streamObject`) and **Next.js 16 + Cache Components** (rate-limit/auth in `proxy.ts`/Node, async dynamic APIs). **Commit to Better Auth** outright and drop Auth.js as a fallback — it's now security-patch-only and its own maintainers point to Better Auth. (§3.2, §3.6, §3.9; §2-A/F)

3. **Re-spec stop/abort and long-run handling around the real resumable-stream semantics.** Document that abort *inverts* when resumability turns on (MVP: `AbortSignal` cancels; P1: dedicated out-of-band stop endpoint + Redis abort channel + reaper as the *primary* path). Evaluate **Vercel Workflows / DurableAgent** as the P1 agent-loop primitive (durable steps solve orphaned runs structurally) — gated behind an adapter to preserve portability (Temporal/Restate as portable equivalents). (§3.3; §1.2, §1.3)

4. **Make cost/abuse protection token-and-dollar aware, not request-count-only.** Add token-rate limits, **per-user rolling cost-budget caps** enforced off `message.cost_usd`, and a **circuit breaker** for runaway tool loops; map to OWASP **LLM10 Unbounded Consumption**. Layer Gateway budgets + **AI Gateway ZDR** (privacy wedge) on top. (§3.4; §1.5, §1.8)

5. **Close the cross-store data-lifecycle gap for GDPR/privacy (the product's core wedge).** Specify cascade **delete/retention across object storage, Redis, and observability traces** (traces are a PII store), with a QStash fan-out delete job and per-store TTLs; harden the `api_key` table (unique per provider, `last_used_at`, rotation/revocation, fresh-DEK-per-record AES-256-GCM); state the guest-BYOK policy (require a real account). (§3.5, §3.7)
